# 📋 决策记录: JWT Secret 外部化

## 🎯 决策概述
**决策编号**: DEC-001  
**决策主题**: JWT Secret 配置管理方式  
**决策时间**: 2026-04-16  
**决策人**: AI助手  
**状态**: ✅ 已实施  

## 🔍 问题背景

### 1. 现状分析
在重构前的代码中，JWT Secret 是硬编码在 `JwtUtil.java` 中的：
```java
private static final String SECRET = "gateway-secret";
```

### 2. 存在的问题
1. **安全风险**: Secret 明文存储在代码中，泄露风险高
2. **部署困难**: 不同环境需要不同的 Secret，硬编码无法灵活配置
3. **维护复杂**: 修改 Secret 需要重新编译和部署
4. **不符合安全最佳实践**: 敏感信息应该外部化配置

### 3. 影响范围
- `common` 模块: JwtUtil 类
- 所有使用 JWT 的服务: service-user, service-order, service-gateway
- 部署和运维流程

## 💡 解决方案选项

### 选项1: 环境变量配置 (✅ 选择)
**方案描述**: 通过环境变量配置 JWT Secret，支持不同环境不同配置

**优点**:
- ✅ 安全性高: Secret 不存储在代码中
- ✅ 灵活性好: 支持不同环境不同配置
- ✅ 部署方便: 无需修改代码即可调整配置
- ✅ 符合云原生最佳实践

**缺点**:
- ❌ 需要额外的配置管理
- ❌ 环境变量管理需要规范

**实施复杂度**: 低

### 选项2: 配置文件外部化
**方案描述**: 通过外部配置文件（如 application.yml）配置

**优点**:
- ✅ 配置集中管理
- ✅ 支持多环境配置文件

**缺点**:
- ❌ 配置文件可能被提交到版本控制
- ❌ 生产环境仍需保护配置文件

**实施复杂度**: 中

### 选项3: 密钥管理服务
**方案描述**: 使用专门的密钥管理服务（如 HashiCorp Vault, AWS Secrets Manager）

**优点**:
- ✅ 安全性最高
- ✅ 支持自动轮换
- ✅ 完善的权限管理

**缺点**:
- ❌ 实施复杂度高
- ❌ 需要额外的基础设施
- ❌ 学习成本高

**实施复杂度**: 高

## 🎯 决策依据

### 1. 安全性考虑
- 环境变量配置可以有效避免 Secret 泄露到代码仓库
- 符合 OWASP 安全最佳实践
- 支持生产环境使用强密码

### 2. 可维护性考虑
- 环境变量配置简单直观
- 易于理解和实施
- 支持现有的部署工具链

### 3. 兼容性考虑
- Spring Boot 原生支持环境变量配置
- 与现有配置体系兼容
- 不影响现有功能

### 4. 成本考虑
- 无需额外基础设施
- 实施成本低
- 维护成本低

## 🛠️ 实施方案

### 1. 代码修改
```java
// 修改前
private static final String SECRET = "gateway-secret";

// 修改后
@Value("${jwt.secret:gateway-secret}")
private String secret;
```

### 2. 配置更新
```yaml
# 在 application.yml 中添加
jwt:
  secret: ${JWT_SECRET:gateway-secret}
  expire: 86400000
```

### 3. 环境变量管理
```bash
# .env.example 示例
JWT_SECRET=your-strong-jwt-secret-key-here

# 生成强密码
openssl rand -base64 32
```

### 4. 部署说明
- 开发环境: 使用默认值或 .env 文件
- 测试环境: 使用环境变量配置
- 生产环境: 必须通过环境变量配置强密码
