# 📋 项目重构规范 (Specification)

## 🎯 项目信息
- **项目名称**: Spring Cloud Alibaba Demo
- **当前版本**: 1.0-SNAPSHOT
- **重构版本**: 2.0-SNAPSHOT
- **负责人**: AI助手
- **创建时间**: 2026-04-16

## 🔍 现状分析

### 1. 项目结构
```
spring-cloud-alibaba-demo/
├── common/                    # 公共模块
├── service-gateway/          # 网关服务
├── service-order/            # 订单服务
├── service-user/             # 用户服务
└── 各种文档文件
```

### 2. 已发现的问题
根据代码审查，发现以下问题：

#### 2.1 🔐 安全性问题
- JWT Secret硬编码 (`gateway-secret`)
- 异常信息可能泄露敏感信息
- 部分Controller直接返回业务字符串

#### 2.2 📝 代码规范问题
- 响应格式不统一（有的用ApiResponse，有的直接返回）
- 日志使用不规范（存在e.printStackTrace()）
- 测试代码中有System.out.println

#### 2.3 🏗️ 架构设计问题
- Demo代码与生产代码混合
- 包结构混乱（按技术分层而非业务领域）
- 缺少API版本控制
- 缺少健康检查和链路追踪

#### 2.4 ⚙️ 配置管理问题
- 部分配置硬编码
- 缺少统一配置管理

## 🎯 重构目标

### 1. 核心目标
- **安全性提升**: 消除安全风险
- **代码规范化**: 统一编码规范
- **架构优化**: 改善项目结构
- **可维护性**: 提高代码可读性和可维护性

### 2. 具体指标
- ✅ 消除所有硬编码的安全信息

- ✅ 完善监控和健康检查

## 🏗️ 技术方案

### 1. 安全性重构
#### 1.1 JWT Secret外部化
```yaml
# 修改前
jwt:
  secret: "gateway-secret"

# 修改后  
jwt:
  secret: ${JWT_SECRET:default-secret}
```

#### 1.2 异常信息统一处理
- 使用自定义异常类
- 统一错误响应格式
- 避免泄露堆栈信息

### 2. 代码规范重构
#### 2.1 响应格式统一
```java
// 所有Controller统一使用
@PostMapping("/api")
public ApiResponse<T> method() {
    return ApiResponse.success(data);
}
```

#### 2.2 日志规范
- 使用SLF4J Logger
- 统一日志格式
- 移除System.out.println

### 3. 架构重构
#### 3.1 包结构重组
```
建议结构：
src/main/java/com/example/service/
├── controller/           # 控制器层
├── service/             # 服务层
├── repository/          # 数据访问层
├── entity/              # 实体类
├── dto/                 # 数据传输对象
├── config/              # 配置类
└── util/                # 工具类（尽量少用）
```

#### 3.2 API版本控制
```
# 当前
/user/login

# 重构后
/api/v1/user/login
```

### 4. 配置管理
#### 4.1 配置外部化
- 使用环境变量
- 支持多环境配置
- 统一配置管理
